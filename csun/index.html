Accessibility is a legal obligation and a moral responsibility

▼

The legal stuff

People don't like to be told what to do - the law is cumbersome and vague. Adherence to "best effort" means that lots of things get missed.

The moral stuff

People like to help people, it makes them feel good.

The accessibility marketplace is only solving for the legal requirements

Because risk management sells

They use words like "compliance" and "auditing," probably because us customers can raise money in the organization to prevent lawsuits, less so for being moral. Organizations lack the capacity to have moral judgement, but people do.

Five Problems with current tools

▼

1

Temporal gap from over-emphasis on reporting

Because reports are periodically done, it means that there can be a month or so between the time content is published and it is finally scanned and then reported on. This not only means that you have content that is published for that period of time, but also means that users have to return to old content to fix it, which is not a good way to learn about accessibility problems, and is cumbersome for the author. While reports are nice with pie charts and stuff, they don't help the author.

2

Reports have no connection to editing process

Reports typically say things like "IMG Tag missing ALT text," which might not be how the user perceives images and alt text in their tool (at CSUMB we call it an image description).

3

Most tools have to "scan" your website

Installing weird software, giving credentials to said software, causes problems. Scanners typically only pick up on served HTML pages, they miss AJAX loaded content.

4

Little to no relationship between CMS and a11y checking

The author should only see issues that they can fix, not problems or false-positives generated by the template.

5

The actual browser experience is ignored

All tools are at best a poor approximation of what a website looks like to an end user. We should be using the browser itself to check content.

QUAIL:

QUAIL Accessibility Information Library

Started in 2009. Used to be PHP, now JavaScript, won't go into details.

▼

Key features

  • 100% JavaScript, based on either jQuery or Sizzle.
  • Can be run on client side as well as server-side with Node.JS or Phantom
  • Supports over 250 tests, has broad test coverage
  • Writing your own tests is easy... if you know JavaScript ;)

Example


$('#content').quail({ guideline : 'wcag2a',
                      testFailed : function(event) { 
                       event.element.addClass('a11y-problem')
                                    .addClass(event.severity)
                                    .before('OMG Fix This!');
                     }
                    });
